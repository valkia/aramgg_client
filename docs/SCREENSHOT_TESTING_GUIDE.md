这是一个自动化测试工具，用于批量测试 `C:\Users\du\.lol-tips-client\screenshots` 目录中的所有截图，验证我们的海克斯卡片检测算法是否准确。

## 使用方法

### 方法 1: 使用 npm 脚本（推荐）

```bash

```

```bash
node electron/test-screenshot-analysis.js
```

| 状态 | 含义 | 说明 |
|------|------|------|
| ✅ 高置信度检测 | 通过 | 检测到 3 张有效卡片，验证通过，置信度 > 90% |
| ⚠️ 检测但置信度低 | 可疑 | 检测到 3 张卡片但验证失败或置信度过低 |
| ⚠️ 部分检测 | 可疑 | 检测到卡片但少于 3 张或验证失败 |
| ❌ 未检测到有效卡片 | 正确拒绝 | 没有检测到有效的海克斯卡片 |
| ❌ 分析失败 | 错误 | 文件损坏或算法崩溃 |

## 工作流程

1. **收集截图**
   ```
   进入游戏 → 海克斯选择界面 → 按 F1 → 截图保存
   ```

2. **运行测试**
   ```bash
   npm run test:screenshots
   ```

3. **分析结果**
   - 查看统计报告
   - 找出误检的截图（部分检测）
   - 找出应该检测但没检测到的截图（未检测）

4. **改进算法**
   - 根据结果调整 HSV 颜色范围
   - 修改验证阈值
   - 重复测试

## 测试场景

### 场景 1: 验证真实海克斯界面

**目标**: 验证在真实游戏中的海克斯选择界面是否被正确识别

**步骤**:
1. 启动游戏
2. 进入海克斯选择阶段
3. 按 F1 截图（或等待自动截图）
4. 进行多张截图（不同角度、光照）
5. 运行 `npm run test:screenshots`
6. 检查统计报告中的"高置信度检测"比例

**预期**:
- 高置信度检测 > 80%
- 所有海克斯截图都应该在"高置信度检测"列表中

### 场景 2: 验证误检修复

**目标**: 验证非海克斯界面是否被正确拒绝

**步骤**:
1. 在应用菜单、游戏菜单、装备商店等界面进行截图
2. 运行 `npm run test:screenshots`
3. 检查统计报告中的"部分检测"和"未检测"

**预期**:
- 非海克斯界面应该在"未检测"列表中（不应该误检）
- 不应该有"部分检测"的截图（或很少）

### 场景 3: 性能测试

**目标**: 验证检测算法的性能

**步骤**:
1. 收集 20+ 张截图
2. 运行 `npm run test:screenshots`
3. 查看统计报告中的"平均耗时"

**预期**:
- 平均耗时 < 300ms
- 最大耗时 < 500ms

## 常见问题

### Q: 没有任何截图可以测试？
**A**: 运行应用，进入游戏海克斯选择界面，按 F1 或等待自动截图。

### Q: 为什么只测试最近 20 张截图？
**A**: 这是为了避免测试时间过长。可以修改脚本中的 `MAX_AUTO_TEST` 常数。

### Q: 测试结果不准确？
**A**: 检查是否：
1. 使用了最新构建的应用（npm run build）
2. 截图是真实游戏截图
3. 应用没有修改算法参数

### Q: 如何改进检测精度？
**A**: 根据测试结果：
1. 查看"部分检测"的截图，调整颜色范围
2. 查看"未检测"的截图，可能需要降低阈值
3. 修改 `electron/image-analyzer.js` 中的参数
4. 重新运行 `npm run build`
5. 重新运行 `npm run test:screenshots` 验证

## 技术细节

### 检测流程

1. **颜色扫描** - 在像素数据中扫描指定颜色范围
2. **区域聚类** - 将相同颜色的像素聚集成区域
3. **边界检测** - 计算每个区域的边界和尺寸
4. **位置验证** - 检查区域是否在屏幕中央
5. **卡片验证** - 检查多个卡片的尺寸和间距一致性
6. **置信度计算** - 基于验证结果计算最终置信度

### 性能特征

- **初始化**: ~50ms (第一次 Sharp 初始化)
- **扫描**: ~100-150ms (像素级扫描)
- **验证**: ~50ms (几何验证)
- **总耗时**: 200-300ms（取决于图像大小和复杂度）
